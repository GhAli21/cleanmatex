## Confirmed problem

## Architecture decision (locked 2026-07-20 — architecture clarification under already-APPROVED D007/D008, not a new decision)

**Current-code research (2026-07-20, full findings preserved so this survives a context reset):**
- `sellGiftCard` (`lib/services/gift-card-service.ts:314-389`) writes `org_gift_cards_mst` + one `org_gift_card_txn_dtl` (`SALE`) row only — no tender, no idempotency key param at all (unlike `redeemGiftCardTx`/`refundGiftCardTx` in the same file, which do accept one). Called from `sellGiftCardAction` (`app/actions/marketing/gift-card-actions.ts:181-222`) ← `gift-card-sell-dialog.tsx` (no method/drawer step in the UI today).
- `topUpWalletTx` (`lib/services/stored-value.service.ts:50-112`) already supports an idempotency-skip and resolves tenant currency correctly (the doc's "OMR hardcode" line item is stale — fixed by B15; the larger no-tender-capture gap is still fully open). Two entry points exist with **two different permission codes**: API route `app/api/v1/customers/[id]/wallet/top-up/route.ts` checks `stored_value:top_up_wallet`; the server action `topUpWallet` in `app/actions/customers/stored-value-actions.ts:160-201` (confirmed — **this is the one the UI actually calls**, from `customer-stored-value-tab.tsx`) checks `stored_value:issue_wallet_credit` (added by B27, meant for the *admin/no-tender* adjustment path). Both currently do the same ledger-only write.
- `issueAdvanceTx` (`lib/services/stored-value.service.ts:212-263`) has **no idempotency key parameter at all**, even though `org_advance_txn_dtl` already has a unique constraint on `(tenant_org_id, idempotency_key)` the service never populates. No dedicated API route exists — only the `issueAdvance` server action.
- The "proven target pattern" referenced in Current evidence (customer-receipt-excess / overpayment-disposition wallet-save path) turned out on inspection **not** to be a real BVM voucher-line pattern — it calls `topUpWalletTx`/`issueAdvanceTx` directly and only backlinks an *existing* RECEIPT voucher id into an `org_fin_overpay_disp_dtl` audit row. The actually-proven `createBizVoucher → addVoucherLine (per leg) → postAndWireBizVoucher → wiring handlers` sequence lives in `collectPaymentTx` (`order-settlement.service.ts:884-1010`, shipped by B4/B5). **No wiring handler exists today that connects a `GIFT_CARD_SALE`/`WALLET_TOPUP`/`CUSTOMER_ADVANCE_RECEIPT` voucher line to anything** — those three `LINE_ROLE` constants and their `LINE_ROLE_REQUIREMENTS` targetTypes (`GIFT_CARD`→GIFT_CARD, `WALLET`→WALLET, `CUSTOMER_ADVANCE_RECEIPT`→**CUSTOMER**, not a separate ADVANCE target type) already exist in `lib/constants/voucher.ts` and are already used by the manual voucher-entry UI (`add-line-dialog.tsx`), but nothing consumes them automatically. **No `GIFT_CARD_RELOAD` transaction type exists in the codebase today** (only an `is_reloadable` flag on the card) — dropped from scope as speculative (CLAUDE.md: don't design for hypothetical requirements); only `GIFT_CARD_SALE` / `WALLET_TOPUP` / `CUSTOMER_ADVANCE_RECEIPT` are in scope.
- The cash-drawer wiring handler (`wiring/cash-drawer-wiring.handler.ts`) gates on `line_role === ORDER_PAYMENT` only; it is structurally reusable (the movement insert already tolerates `order_id: null`) but its comments/naming are order-specific — a **separate** stored-value cash-drawer handler is cleaner than broadening the order one.
- `collectPaymentTx`'s idempotency pattern (`org_idempotency_keys`, resource_type `'collect_payment'`, check-first/compute/store-result, `order-settlement.service.ts:674-1053`) is the exact mechanism to replicate here with `resource_type: 'stored_value_funding'`.

**Locked design v2 (revised 2026-07-20, same session — a second external-proposal review caught a real simplification before any migration was written; see "Revision v2" note below):**

1. **One new table** (existing wallet/advance/gift-card ledger tables, `org_order_credit_apps_dtl`, and BVM voucher tables all keep their current roles — nothing renamed or replaced):
   - **`org_sv_funding_tenders_dtl`** — one row per real-money tender leg: `id`, `tenant_org_id`, `fin_voucher_id` (FK — **the voucher IS the funding aggregate**, see Revision v2 note), `fin_voucher_trx_line_id` (unique — wiring lineage), `funding_type` (`GIFT_CARD_SALE`/`WALLET_TOPUP`/`CUSTOMER_ADVANCE_RECEIPT`, denormalized from the line at wire time so the table is self-sufficient for reporting), `target_type`/`target_id` (mirrors `TARGET_TYPE`, denormalized from the line for the same reason), `leg_index`, `payment_method_code`, `org_payment_method_id`, `amount`, `tendered_amount`/`change_returned_amount` (cash legs — denormalized from the voucher line, same convention `org_order_payments_dtl` already follows), `currency_code`, `status` (`COMPLETED`/`FAILED`/`REVERSED` — row-level lifecycle), `payment_status` (Revision v3 — the canonical resolved status the leg cleared at, e.g. `COMPLETED`; recorded for audit even though only `ORDER_PAYMENT_LIFECYCLE_STATUSES.COMPLETED`-set legs are ever accepted in v1), `cash_drawer_session_id`, `pos_session_id`, `gateway_code`/`gateway_reference`, `check_number`/`check_bank`/`check_date`, `idempotency_key` (unique per tenant), `confirmed_at`/`failed_at`/`reversed_at`, audit columns.
   - **Rejected:** reusing `org_order_payments_dtl` with `order_id = NULL` (conflates order-scoped and non-order-scoped payment facts) and reviving `org_payments_dtl_tr` (retired platform-wide, never touch it again). **Also rejected (v2): a separate `org_sv_fundings_mst` funding-header table** — see Revision v2 note.

2. **Shared `stored-value-funding.service.ts`** (new), one function per funding type or one parameterized `fundStoredValue()` — synchronous confirmed-tender flow only (today's real scope: cash/card-swipe entered by an operator, confirmed at the moment of entry — **no online payment-gateway callback integration exists for any of these three entry points today**, so a full async PENDING→gateway-callback state machine is **deferred, not built**, until a real gateway integration for stored-value funding is scoped):
   1. Validate tenant/branch/customer/currency (resolve-or-throw, no literals — reuse `lib/money/currency-resolution.ts` from B15).
   2. Idempotency check-first (`org_idempotency_keys`, resource_type `stored_value_funding`) — replicates `collectPaymentTx`'s exact pattern.
   3. For `GIFT_CARD_SALE` only: create the `org_gift_cards_mst` row first, in `GENERATED` status (existing pre-activation status, reused as-is — the card is **unfunded** until tender confirms; `original_amount`/`current_balance`/`available_amount` start at 0). This gives the voucher line a real `target_id` to point at before any money is confirmed.
   4. `createBizVoucher` (RECEIPT, `total_amount` = the intended funded amount — this is the funding's `funded_amount`, no separate header field needed) → `addVoucherLine` per tender leg (role = `GIFT_CARD_SALE`/`WALLET_TOPUP`/`CUSTOMER_ADVANCE_RECEIPT`, target per `LINE_ROLE_REQUIREMENTS`) → `postAndWireBizVoucher`.
   5. New wiring handlers (below) create the tender-leg rows, then the finalizer sums them, credits the correct existing ledger (`topUpWalletTx`/`issueAdvanceTx`/the gift-card sale ledger insert, which also flips the card `GENERATED → ACTIVE` for a sale) exactly once, and creates drawer movements for cash legs.
   6. Emit a plain completion outbox event (same pattern as every other financial write path — `PAYMENT_RECEIVED`, `VOUCHER_POSTED_AND_WIRED`); store idempotency result; return. GL/liability posting itself stays B6's job (see point 6 below) — this package only emits the fact that funding completed.
   - All steps run inside the one Prisma transaction `collectPaymentTx` already demonstrates is workable for this shape of multi-write operation.

3. **Two new wiring handlers** (registered in `voucher-wiring.service.ts`'s `WIRING_HANDLERS`, after the order-payment ones):
   - **`stored-value-funding-wiring.handler.ts`** — `canHandle`: `line_role` ∈ `{GIFT_CARD_SALE, WALLET_TOPUP, CUSTOMER_ADVANCE_RECEIPT}`. Creates the `org_sv_funding_tenders_dtl` row keyed on `fin_voucher_trx_line_id` (replay-safe: same line → same result, conflicting replay on an already-wired line → reject), writes the new tender-leg id back onto the voucher line in-memory (`line.sv_funding_tender_id`, mirroring how `order_payment_id` is handed to the cash-drawer handler) and persists it to a new nullable `org_fin_voucher_trx_lines_dtl.sv_funding_tender_id` column, then calls an idempotent `finalizeStoredValueFundingIfReady()` that: locks the target row, sums confirmed tender legs for `fin_voucher_id = voucherId`, asserts the total equals the voucher's own `total_amount`, credits the ledger exactly once (guarded by checking no ledger row already carries this `fin_voucher_id`), links it back to the voucher, and (for gift cards) activates the card.
   - **`stored-value-cash-drawer-wiring.handler.ts`** — mirrors `cash-drawer-wiring.handler.ts`'s movement-creation body but keyed off the new line roles instead of `ORDER_PAYMENT`, linking `funding_tender_id` (reads `line.sv_funding_tender_id` set by the handler above) instead of `order_id`/`order_payment_id`. New nullable `org_cash_drawer_movements_dtl.funding_tender_id` FK column; new `sys_cash_drawer_movement_type_cd` code `SV_FUNDING_TENDER` (`default_direction='IN'`, `affects_expected_cash=TRUE`). Verified against B16/B35's unified expected-cash formula (`lib/services/cash-drawer-cash-facts.ts` + `cash-drawer.service.ts`): the movement term only excludes rows where `order_payment_id IS NOT NULL` (sale-mirror movements already counted via the order-payment-ledger term); a `SV_FUNDING_TENDER` row has `order_payment_id = NULL` and no other term double-counts it, so it is correctly picked up with **no change needed to B16/B35's drawer-close code**. Kept **separate** from the order handler rather than broadening its `canHandle` — the existing handler's comments/shape are order-specific and a second focused handler is clearer than overloading one.

4. **Defect fixes bundled into this package** (each was independently already a confirmed gap, not new scope creep):
   - `sellGiftCard` gains `idempotencyKey` (matching `redeemGiftCardTx`/`refundGiftCardTx` in the same file) and routes through the shared funding service instead of activating the card before a tender is confirmed.
   - `issueAdvanceTx` gains an `idempotencyKey` param, finally using the unique constraint `org_advance_txn_dtl` already has.
   - New sparse unique index on `org_gift_card_txn_dtl (tenant_org_id, idempotency_key) WHERE idempotency_key IS NOT NULL` — the one ledger table of the three with no such protection today.
   - **Permission semantics frozen, no code renamed:** `stored_value:top_up_wallet` = customer-funded top-up requiring real tender (the new funding-service path); `stored_value:issue_wallet_credit` = administrative/promotional credit with **no** tender fact (B27's existing `topUpWallet` server-action gate, left as-is for that use case). The UI's real-money top-up flow moves to call the funding service (and therefore `stored_value:top_up_wallet`); an admin-adjustment surface (if/when one exists as a distinct screen) keeps `issue_wallet_credit`. For advance receipt: **kept the existing `stored_value:issue_advance` code** rather than adding a new `stored_value:receive_advance` — avoids an unnecessary permission migration for a purely semantic clarification.

5. **Reversal — contract defined, UI deferred** (matches this doc's own established pattern for out-of-scope-but-acknowledged items, e.g. expiry/breakage deferred to B19/B25): a funding reversal references the original voucher (BVM `REVERSED`/`PARTIALLY_REVERSED` lineage — reusing the voucher's own reversal fields instead of a bespoke self-FK), is idempotent, debits the exact ledger, reverses the drawer movement for cash legs, and is **blocked** if any of the funded value has already been redeemed/transferred (routes to a governed adjustment process instead). No reversal UI/API ships in this package — the contract exists so a later package (or an emergency fix) doesn't have to invent one under pressure.

6. **Rejected/trimmed from external proposals** (two separate reviews this session — with reasons, so this isn't silently different from what was reviewed): a full GL liability-posting event/handler — genuinely belongs to B6 (ERP order-to-cash wiring), which already owns GL posting for this domain; building liability-posting logic here would duplicate B6's future work (a plain completion outbox event is still emitted, per point 2 above — that's an existing pattern every financial write path already follows, not liability posting). `GIFT_CARD_RELOAD` funding type — no such transaction exists in the codebase today (see research above); re-adding it without a real reload flow existing would be speculative scope creep. Full async gateway-callback state machine — no gateway integration exists for these 3 entry points today.

**Revision v2 note (same session, before any migration was written):** a second external-architecture review correctly pointed out that the funding-header table (`org_sv_fundings_mst`) from the first pass was solving a problem this design doesn't have — its only real job was tracking a `PROCESSING` state *across transactions*, but the whole funding flow is one synchronous Prisma transaction (no async gateway callback exists for these 3 entry points), so nothing ever observes that intermediate state from outside the transaction. The BVM voucher this design already creates has `id`/`total_amount`/`voucher_status`/`currency_code`/reversal fields — everything the header table would have duplicated. Dropping it removes an entire table and its bookkeeping with zero loss of capability (verified: `funded_amount` = voucher `total_amount`, same pattern `collectPaymentTx` already uses for `totalCollected`; `target_type`/`target_id` were already going to live on the voucher lines). Two other suggestions from that same external review were evaluated and **rejected** — see point 6. The proposed table name `org_stored_value_funding_tenders_dtl` (37 chars) was also rejected for exceeding CLAUDE.md's 30-char DB object name limit; kept the already-chosen `org_sv_funding_tenders_dtl` (27 chars).

**Revision v3 note (same session, a third external review — before the migration draft was finalized; each point below independently checked against the live schema/code, not taken on faith):**
- **Voucher aggregate invariant, now frozen explicitly:** a funding voucher created by this service always has exactly one funding role, one target (one wallet/card/advance), one customer (where applicable), one currency, and one or more tender legs. Nothing in this package ever mixes funding lines with other line types on the same voucher, or funds two different targets from one voucher. This was already true by construction (the service only ever creates single-purpose vouchers) but is now stated as a binding rule for anyone extending this later.
- **Manual voucher entry is a real bypass path once these line roles are wired — closed for v1:** once `stored-value-funding-wiring.handler.ts` is registered, ANY voucher line with role `GIFT_CARD_SALE`/`WALLET_TOPUP`/`CUSTOMER_ADVANCE_RECEIPT` — including one created through the generic manual voucher-entry UI (`add-line-dialog.tsx`), which already offers these roles today — triggers real ledger credit. That UI has none of the funding service's guards (currency resolution, D9 method-availability check, gift-card-must-exist-first sequencing, single-currency-across-legs assertion). **v1 fix: hide these three roles from the generic manual voucher-entry role picker.** Re-enable only if that UI is ever upgraded to the full governed tender workflow. Added to Scope/Delivery surfaces below.
- **Finalization timing — checked, not a bug, but stated explicitly:** `finalizeStoredValueFundingIfReady()` finalizes only when `SUM(confirmed tender legs for this voucher) = voucher.total_amount`. Because every leg amount is `> 0` (DB CHECK) and the voucher's `total_amount` is set at creation to the exact sum of all intended legs (same invariant `collectPaymentTx` relies on for `totalCollected`), that sum cannot reach the total until every leg has been wired — there is no window where a partial set of legs can prematurely equal the total. No structural change needed; stated here so a future reader doesn't have to re-derive it. **New validation added:** the finalizer also asserts every confirmed leg shares one `currency_code` (part of the frozen invariant above) before crediting.
- **DIRECT_TENDER is the only source this package creates a tender fact for — frozen non-goal:** the existing overpayment-disposition / receipt-excess wallet-save path (`executeOverpaymentDispositionTx`) calls `topUpWalletTx`/`issueAdvanceTx` directly and never creates a `WALLET_TOPUP`/`CUSTOMER_ADVANCE_RECEIPT`-role voucher line (verified in the current-code research above) — so the new wiring handler never fires for it, and this package does not touch that path. A future implementer must not route receipt-excess, overpayment disposition, refund-to-wallet, or the admin no-tender adjustment through these same line roles; each of those is a different source category (no new tender fact, no new voucher, no drawer movement) and must stay on its own existing path.
- **Distinguishing signal for reporting/reconciliation:** on `org_wallet_txn_dtl`/`org_advance_txn_dtl`/`org_gift_card_txn_dtl`, `fin_voucher_id IS NOT NULL` is now the authoritative signal that a ledger row is customer-funded DIRECT_TENDER (came through this package's governed path); `fin_voucher_id IS NULL` means administrative/no-tender (B27's existing admin-adjustment path). No new column needed — both columns already exist on all three ledger tables.
- **Payment-status gate, a real gap this review caught:** the original design didn't say what happens if a tender leg's payment method resolves to a non-immediately-COMPLETED status (e.g. `CHECK`/`BANK_TRANSFER`, which `resolvePaymentStatus()` starts as `PENDING`). Since this package deliberately ships no async completion mechanism (no gateway callback, no later "confirm this leg" step), silently crediting a stored-value balance against a still-pending tender would be a real money-safety gap, and silently accepting it with no way to ever complete it would leave a stuck funding voucher. **Fix: the service validates every leg's resolved status up front (reusing `ORDER_PAYMENT_LIFECYCLE_STATUSES.COMPLETED` = `['COMPLETED','CAPTURED','SETTLED']`, the same canonical set `cash-drawer-cash-facts.ts` already uses) and rejects the request outright if any leg would resolve outside that set** — v1 stored-value funding only accepts payment methods that settle immediately (cash, card-present). `CHECK`/`BANK_TRANSFER`/gateway methods stay unavailable for these 3 entry points until an async completion path is designed (tracked, not built here).
- **Outbox event named:** `OUTBOX_EVENT_TYPES.STORED_VALUE_FUNDING_COMPLETED` (new constant, same enum as `PAYMENT_RECEIVED`/`VOUCHER_POSTED_AND_WIRED`), payload: `tenantOrgId`, `branchId`, `voucherId`, `fundingType`, `targetType`/`targetId`, `customerId`, `currencyCode`, `fundedAmount`, tender-leg summary (method/amount per leg), `cashDrawerSessionId`/`posSessionId`, `completedAt`, `idempotencyKey`. Consumed later by B6 for GL liability posting — this package only emits the fact.
- **Rollback corrected — must not reopen the money-capture gap:** the original Rollback line said disabling the flag lets the 3 entry points fall back to calling `topUpWalletTx`/`issueAdvanceTx`/`sellGiftCard` directly again — that resumes the exact tender-less gap this package exists to close (C3), which is unsafe as a *default* rollback behavior. **Corrected: disabling the flag blocks the 3 real-money entry points (shows an unavailable/maintenance state) rather than silently resuming tender-less writes.** The legacy functions stay in the codebase as the funding service's own building blocks (not deleted, still directly callable by code that has its own governance, e.g. the admin no-tender path) — they are just not reachable from these 3 UI entry points once this ships, flag on or off.
- **Ledger-primitive hardening — evaluated, deferred (not adopted now):** a suggestion to require every `topUpWalletTx`/`issueAdvanceTx`/`sellGiftCard` caller to pass an explicit "source" tag, or split them into per-source internal functions. The `fin_voucher_id IS NOT NULL` signal (above) already distinguishes governed-voucher writes from admin ones without touching existing call sites. Refactoring the receipt-excess/refund-to-wallet/admin-adjustment call sites is out of this package's scope (Remediation_Work_Packages CLAUDE.md: "keep each Bxx package focused on its own responsibility, do not silently merge unrelated work") — noted here as a future hardening candidate, not built.
- **Composite (tenant-qualified) FKs — evaluated, not adopted:** checked migration 0303, which is the precedent for exactly this kind of voucher back-link column (`org_wallet_txn_dtl.fin_voucher_id`, `org_order_payments_dtl.fin_voucher_id`, etc.) — none of those use a composite `(tenant_org_id, id)`-qualified FK, they're plain single-column FKs (or no FK at all) with tenant isolation enforced by RLS + `tenant_org_id NOT NULL` on the row itself, not by the FK shape. `org_sv_funding_tenders_dtl.fin_voucher_id` keeps a single-column FK for consistency with that established convention (already stricter than several of its precedents, which have no FK at all).

## Required outcome
Every funding operation records tender (payment fact incl. method/status via `org_sv_funding_tenders_dtl`), a BVM receipt voucher line (GIFT_CARD_SALE / WALLET_TOPUP / CUSTOMER_ADVANCE_RECEIPT roles), drawer movement for drawer-required cash, all keyed (D010), currency from tenant resolution.

## Scope
New `org_sv_funding_tenders_dtl` table — tender/amount/status/`tendered_amount`/`change_returned_amount`/canonical `payment_status` columns (the BVM RECEIPT voucher created for each funding event is the funding aggregate — no separate header table, see Revision v2); shared `stored-value-funding.service.ts`; two new wiring handlers (`stored-value-funding-wiring.handler.ts`, `stored-value-cash-drawer-wiring.handler.ts`); retrofit `sellGiftCard`/`topUpWalletTx`/`issueAdvanceTx` call sites (GC-sale action, wallet top-up route/action, advance issue action) to route through the shared service; idempotency keys added to `sellGiftCard`/`issueAdvanceTx`; sparse unique index on `org_gift_card_txn_dtl`; new nullable `sv_funding_tender_id` column on `org_fin_voucher_trx_lines_dtl` and `funding_tender_id` column on `org_cash_drawer_movements_dtl`; new `SV_FUNDING_TENDER` cash-drawer movement type; new `STORED_VALUE_FUNDING_COMPLETED` outbox event type; hide `GIFT_CARD_SALE`/`WALLET_TOPUP`/`CUSTOMER_ADVANCE_RECEIPT` roles in the generic manual voucher-entry UI (Revision v3 — closes a bypass path); permission semantics frozen (no new codes); tests.
**Frontend surface (rule 7):** GC-sale dialog, wallet top-up screen, and advance-receipt UI gain a tender step (method, amount, drawer session where required) and a funding receipt/print — funding without choosing how the money arrived becomes impossible in the UI as well as the API.

## Out of scope
GL liability journals (B6/B25 — not duplicated here); expiry/breakage (B19/B25); promotional credit design (D008 scope note); `GIFT_CARD_RELOAD` (no such transaction exists yet — added only if a real reload flow is ever built); async gateway-callback funding (no gateway integration exists for these entry points today); reversal UI/API (contract defined, not built).

## End-to-end operational flow

1. Operator opens GC-sale / top-up / advance dialog, enters amount → tender step requires method (+ cash tendered/drawer when cash) → submit with idempotency key.
2. Service checks `org_idempotency_keys` first (replay-safe); for a GC sale, creates the `org_gift_cards_mst` row `GENERATED`/unfunded first; creates the RECEIPT voucher (`total_amount` = intended funded amount) → `addVoucherLine` per tender leg → `postAndWireBizVoucher`.
3. Wiring creates `org_sv_funding_tenders_dtl` row(s), a drawer movement for cash legs, then the finalizer sums confirmed tenders, asserts the total equals the voucher's `total_amount`, credits the existing wallet/advance/gift-card ledger exactly once (activating the card for a GC sale), and emits the completion outbox event.
4. UI shows success with funding receipt link; duplicate submit replays the original cached result (no double-credit).
5. Reconciliation links ledger ↔ voucher ↔ tender; missing-link or total-mismatch rows surface as blockers; failure mid-tx rolls back every fact (voucher, tender rows, ledger credit, drawer movement, gift-card row) together.

UI states: standard Cmx state contract — loading, empty, validation errors, permission-denied (action disabled with reason), duplicate-click protection (in-flight disable + idempotent replay), processing, success, retry on failure; funding history visible on the card/wallet/advance record.
