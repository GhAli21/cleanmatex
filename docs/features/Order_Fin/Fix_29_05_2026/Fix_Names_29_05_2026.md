# 2. Final naming decisions

## 2.1 Order total naming

Rename conceptually:

```text
finalTotal → saleTotal
total → total_amount / sale_total_amount
```

Meaning:

```text
Full sale/service value after commercial discounts, tax, and rounding.
Does not subtract gift card, wallet, advance, credit note, customer credit, or real payments.
```

Recommended code name:

```ts
saleTotal
```

Recommended DB name:

```sql
total_amount
```

Keep old `org_orders_mst.total` only temporarily as legacy/backward compatibility.

---

## 2.2 Gift card / stored-value naming

Rename conceptually:

```text
giftCardApplied → giftCardCreditAppliedAmount
```

Better generic name:

```ts
storedValueCreditAppliedAmount
```

But if specifically gift card:

```ts
giftCardCreditAppliedAmount
```

Meaning:

```text
Gift card amount applied as stored-value credit settlement.
Not discount.
Not payment.
Not pricing reduction.
```

DB canonical:

```sql
total_credit_applied_amount
```

Legacy:

```sql
gift_card_applied_amount
```

should become deprecated/display-only.

---

## 2.3 Paid amount naming

Rename ambiguous:

```text
paidAmount → completedRealPaymentAmount
```

or for order summary:

do not add total_completed_payment_amount as a separate DB column. Keep total_paid_amount as the canonical DB column and use totalCompletedPaymentAmount only in TypeScript/API naming.

```text
total_paid_amount → keep it same → total_paid_amount
```

But since `total_paid_amount` already exists and is understandable, keep DB as:

```sql
total_paid_amount
```

Use clearer code variable:

```ts
totalCompletedPaymentAmount
```

Meaning:

```text
Completed real payments only.
Cash/card/bank/check/mobile/gateway after confirmation.
Does not include gift card/wallet/advance/credit note/customer credit.
```

---

## 2.4 Credit amount naming

Use:

```ts
totalCreditAppliedAmount
```

DB:

```sql
total_credit_applied_amount
```

Meaning:

```text
Applied stored-value/customer-credit settlement.
```

Includes:

```text
GIFT_CARD
WALLET
CUSTOMER_ADVANCE
CREDIT_NOTE
CUSTOMER_CREDIT
LOYALTY_VALUE
MANUAL_CREDIT
```

---

## 2.5 Outstanding amount naming

Keep:

```ts
outstandingAmount
```

DB:

```sql
outstanding_amount
```

Meaning:

```text
Remaining unsettled amount after completed real payments and applied credits.
```

Formula:

```text
outstanding_amount =
  total_amount
- total_paid_amount
- total_credit_applied_amount
```

---

## 2.6 AR receivable naming

Rename:

```text
netReceivableAmount → arReceivableAmount
net_receivable_amount → ar_receivable_amount
```

Meaning:

```text
Amount that belongs to AR/customer receivable.
Only for CREDIT_INVOICE / B2B / INVOICE.
```

Rule:

```text
PAY_ON_COLLECTION → ar_receivable_amount = 0
Cash/card/mobile/gateway fully paid → ar_receivable_amount = 0
CREDIT_INVOICE/B2B/INVOICE → ar_receivable_amount = outstanding_amount
```

---

## 2.7 Pay on collection naming

Keep:

```ts
payOnCollectionAmount
```

DB:

```sql
pay_on_collection_amount
```

Meaning:

```text
Retail operational outstanding to be collected at pickup/delivery.
Not AR.
```

---

## 2.8 Tax naming

Canonical DB:

```sql
total_tax_amount
```

Recommended code:

```ts
totalTaxAmount
```

Legacy/deprecated:

```sql
tax
vat_amount
```

Rule:

```text
Do not add tax + vat_amount + total_tax_amount.
Use total_tax_amount as canonical.
```

---

## 2.9 Discount naming

Canonical DB:

```sql
total_discount_amount
```

Recommended code:

```ts
totalCommercialDiscountAmount
```

or:

```ts
totalDiscountAmount
```

Legacy/deprecated:

```sql
discount
promo_discount_amount
```

Rule:

```text
Gift card is not discount.
Wallet/advance/credit note/customer credit are not discounts.
```

# 4. Database migration instructions

## Phase DB-1 — Add canonical columns, do not drop yet

Create migration:

```text
supabase/migrations/<next>_canonical_order_financial_column_names.sql
```

SQL:

```sql
begin;

alter table public.org_orders_mst
  add column if not exists total_amount numeric(19,4) not null default 0,
  add column if not exists subtotal_amount numeric(19,4) not null default 0,
  add column if not exists items_base_amount numeric(19,4) not null default 0,
  add column if not exists total_paid_amount numeric(19,4) null,
  add column if not exists ar_receivable_amount numeric(19,4) not null default 0;

-- Rename net_receivable_amount if it exists and ar_receivable_amount does not contain real data yet.
do $$
begin
  if exists (
    select 1 from information_schema.columns
    where table_schema = 'public'
      and table_name = 'org_orders_mst'
      and column_name = 'net_receivable_amount'
  ) then
    update public.org_orders_mst
    set ar_receivable_amount = coalesce(ar_receivable_amount, net_receivable_amount, 0)
    where coalesce(ar_receivable_amount, 0) = 0;
  end if;
end $$;

-- Backfill canonical fields from current fields.
update public.org_orders_mst
set
  subtotal_amount = coalesce(subtotal_amount, 0) + case when coalesce(subtotal_amount,0)=0 then coalesce(subtotal,0) else 0 end,
  items_base_amount = coalesce(items_base_amount, 0) + case when coalesce(items_base_amount,0)=0 then coalesce(subtotal,0) else 0 end,
  total_amount = case
    when coalesce(total_amount, 0) > 0 then total_amount
    else coalesce(total, 0)
  end,
  ar_receivable_amount = coalesce(ar_receivable_amount, 0);

commit;
```

Important: if `total` is known polluted for gift-card orders, do **not** trust this backfill for those rows. Run the gift-card repair migration after code fix.

# 10. Final instruction to Codex

Use this exact instruction:

```text
Rename ambiguous financial variables and DB/API fields to canonical Order Fin names.
- You Must add db comments for each column to clear describe it.
- Also in code documentations and file documentations.
 
The target semantics are:
- saleTotal / total_amount / orderTotalAmount = full sale value after commercial discounts, tax, and rounding; before payments and stored-value credits.
- totalCompletedPaymentAmount / total_paid_amount = completed real payments only.
- totalCreditAppliedAmount / total_credit_applied_amount = gift card, wallet, advance, credit note, customer credit, loyalty value.
- outstandingAmount / outstanding_amount = sale total - completed payments - applied credits.
- arReceivableAmount / ar_receivable_amount = AR receivable amount only for CREDIT_INVOICE/B2B/INVOICE.
- payOnCollectionAmount / pay_on_collection_amount = retail operational collection amount, not AR.
- totalCommercialDiscountAmount / total_discount_amount = commercial discounts only.
- totalTaxAmount / total_tax_amount = canonical tax summary.

Do not keep or introduce ambiguous names like finalTotal, amountDue, netReceivable, invoiceAmount, or paidAmount unless they are legacy compatibility and clearly marked deprecated.

Do this in phases:
1. Add canonical DB/API fields.
2. Refactor TypeScript names.
3. Backfill and repair gift-card affected orders.
4. Update UI mapper and tests.
5. Keep legacy fields temporarily.
6. Drop legacy fields in a later migration after code search and validation.
```

This gives you a clean naming model without breaking production in one risky migration.
